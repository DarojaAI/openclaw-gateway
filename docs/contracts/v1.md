# Contract Version v1

This document describes the obligations that bind an agent declaring
`contract_version: v1` in its `.openclaw/agent-config.yaml`. The contract
is enforced at the gateway (Layer 3b), not at the agent itself; agents
are expected to behave well, but the gateway is the final arbiter.

## Bindings

1. The agent resolves `@handle` lookups against the gateway's `handle_id`
   registry at gateway boot. The gateway mints a UUID v4 `handle_id` on
   first deploy and binds it to the lowercase `@handle`; renames of the
   `@handle` do not require re-minting the `handle_id`, but the registry
   is the authoritative source for routing decisions.

2. The agent accepts `@A ask @B` bridge calls as a single-turn response
   and never bridges again within the same call chain. A bridge is one
   `ask` hop from another agent; the second hop is rejected at the
   gateway with HTTP 4xx regardless of the agent's internal policy. This
   is what `max_bridge_depth: 1` enforces.

3. The agent advertises its `capabilities` array and accepts
   capability-based dispatch when `@handle` is omitted. Capability
   strings are lowercase, hyphen-separated, and unique within the
   agent's config. The gateway routes by capability tag when the
   caller does not name a specific handle.

4. The agent honors `allowed_channels` as Discord snowflake IDs (not
   display names). Rejects routing outside the set; in dry-run mode logs
   the rejection, in enforcement mode refuses. Snowflakes are 17-20
   digits; display names and channel @mentions are not accepted as
   routing keys.

5. The agent responds to gateway heartbeat pings within
   `2 * interval_hours`. Missing two pongs in a row triggers quarantine.
   Heartbeats are disabled by default (`heartbeat.enabled: false`); an
   agent that opts in accepts the liveness obligation.

6. The agent's `max_bridge_depth` is enforced at the gateway, not at
   agent policy. Gateway rejects depth violations with 4xx. The v1
   contract permits only `{0, 1}` — `0` disables bridges entirely, `1`
   is the single-bridge default.

**Bumping to v2 requires a written migration in docs/contracts/v2.md; v1 agents keep working.**
